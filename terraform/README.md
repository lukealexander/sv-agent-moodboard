# terraform/

Thin per-service Terraform that deploys this service's full stack — one API
(`apps/api`) and one frontend (`apps/frontend`) — onto the shared Supervenient
AI platform by composing the `deployment` modules.

This root does **not** create shared infrastructure. The platform (VPC, ALB,
ECS cluster, Cognito, RDS, DNS) is provisioned once by the `infrastructure`
repo and published to SSM under `/<environment>/platform/*`; the modules read
it from there.

## Layout

| File                      | Purpose                                                             |
| ------------------------- | ------------------------------------------------------------------- |
| `main.tf`                 | `module "api"` + `module "frontend"` calls, and `module "db"` (gated by `create_database`). |
| `variables.tf`            | Per-service inputs (`service_name`, `github_repo`, …).              |
| `outputs.tf`              | Values `scripts/deploy.sh` reads (ECR URL, ECS service, …).         |
| `providers.tf`            | AWS provider, region `eu-west-2`. No `backend` block (see below).   |
| `terraform.tfvars.example`| Copy to `terraform.tfvars` (gitignored) and fill in.               |
| `modules/`                | Git submodule → `Supervenient-AI/deployment` (the `api`/`frontend` modules). |
| `backend.tf`              | Generated by `scripts/bootstrap-state.sh` (S3 + DynamoDB). Not committed. |

## Modules submodule

`modules/` is a git submodule pinned to the `deployment` repo. After cloning
this service repo:

```bash
git submodule update --init terraform/modules
```

> **TODO:** the submodule currently tracks `deployment`'s default branch
> (`main`). Once that repo cuts a release tag, pin to it:
> `cd terraform/modules && git checkout v1.0.0 && cd - && git add terraform/modules`.

CI (`actions/checkout`) must set `submodules: recursive` (and, for a private
`deployment` repo, a token with read access).

## Usage

State backend is per-service; create it once, then deploy via the script:

```bash
SERVICE_NAME=my-app ./scripts/bootstrap-state.sh   # writes terraform/backend.tf
cd terraform && terraform init                      # initialises the S3 backend
```

Then use [`scripts/deploy.sh`](../scripts/deploy.sh) for releases — it handles
the ECR bootstrap ordering, the ARM64 image build/push, the explicit ECS
rollout, and the Amplify build trigger. See
[docs/infrastructure.md](../docs/infrastructure.md) for the full flow.

## Database (optional)

A managed Postgres database on the shared RDS instance is **off by default**.
Turn it on with a single switch in `terraform.tfvars`:

```hcl
create_database = true   # default: false
```

When `true`, `terraform apply` creates the `db` module's **plumbing** — an
(empty) Secrets Manager secret for `DATABASE_URL`, a scoped bootstrap IAM role,
and a one-shot Fargate task definition — and the `api` module is wired to read
that secret. The database itself is **not** created by `apply`: on each deploy,
[`scripts/deploy.sh`](../scripts/deploy.sh) runs the one-shot bootstrap task
(`aws ecs run-task`) that idempotently creates the `<service>` database and
`<service>_user` role on the shared RDS, writes the connection string into the
secret, and runs `alembic upgrade head` — before rolling out the new API code.

When `false` (the default), the `db` module is dropped entirely (`count = 0`),
no DB resources exist, deploy.sh skips the bootstrap step, and the API runs
stateless with no `DATABASE_URL` injected.

**Bring-your-own database:** to point the API at an externally managed database
instead, leave `create_database = false` and set `database_url_secret_arn` to a
Secrets Manager ARN you own. It takes precedence over `create_database`.
